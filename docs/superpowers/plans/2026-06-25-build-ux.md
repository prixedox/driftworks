## Design decisions (locked)

### Erase refund
- Refund rate: **100%** of the module's build cost is returned to `inventory` on erase. Rationale: ROADMAP §6.1 says "cheap/free erase refunds most build cost"; 100% is the most forgiving and easiest to tune later via a single constant.
- A private constant `ERASE_REFUND = 1.0` (integer multiply: `Math.round(cost * ERASE_REFUND)`) lives in `world.ts`. Since costs are integers and refund is 100%, the result is always exactly the cost — no rounding needed. If the refund rate is changed to a fraction later, the `Math.round` is already in place.
- The existing `remove(c)` method is **extended** (not replaced): it now also refunds and optionally pushes to the undo log. The `Command` `{ type: 'remove', cell }` already exists and continues to work — no new command type for erase refund.

### Undo
- The undo stack lives in `World` to keep it deterministic and replayable.
- Stack capacity: **32 entries** (constant `UNDO_LIMIT = 32` in `world.ts`). Oldest entries are discarded when the limit is reached (FIFO overflow).
- An `UndoEntry` is a discriminated union:
  ```ts
  type UndoEntry =
    | { op: 'place'; cell: number; type: ModuleType; dir: Dir }
    | { op: 'remove'; cell: number; type: ModuleType; dir: Dir };
  ```
  Place pushes a `{ op: 'place', ... }` entry so undo can reverse it (remove the module, refund nothing — the cost was already deducted). Remove pushes a `{ op: 'remove', ... }` entry so undo can reverse it (restore the module at zero cost — the refund was already given).
- A new command `{ type: 'undo' }` is added to the `Command` union. `world.undo()` pops the top entry and applies the inverse. Undo itself does **not** push to the undo log (non-recursive, matching common game conventions).
- `placeRaw()` is not logged (used by `loadDemo`/`loadSave`/paste internal). `place()` logs. `remove()` logs (unless an internal flag `skipLog` is set — used by paste's "undo remove" logic, which never applies here, but the flag allows clean composition).

### Blueprint copy/paste
- **Tool:** A new tool `'blueprint'` is added to the `Tool` type and the hotbar. It is always unlocked (no research gate, no cost).
- **Selection:** the main thread tracks two corner cells (`bpCorner1`, `bpCorner2`) during a drag with the Blueprint tool. After pointer-up the selection rectangle is computed client-side, the clipboard is populated by sending a `{ type: 'blueprint'; action: 'copy'; cells: number[] }` command, which reads the current module layout for those cells and stores a clipboard in `World`.
- **Clipboard:** `World.clipboard: { relCol: number; relRow: number; type: ModuleType; dir: Dir }[]` stores the relative offsets from the top-left corner of the selection rectangle. It is reset on each copy.
- **Ghost preview:** while hovering after a copy, the main thread renders a multi-cell ghost overlay by iterating `snapshot.clipboard` (new field on `Snapshot`) and calling `renderer.setGhost` once per cell (or a new `renderer.setBlueprintGhost(cells)` if the renderer supports it — see Task C3).
- **Paste command:** `{ type: 'blueprint'; action: 'paste'; originCell: number }`. `world.paste(originCell)` iterates the clipboard, computes absolute cells, and for each entry: skips if occupied, skips if unaffordable (checks partial costs accumulated so far against inventory), deducts cost and places. Cells are processed in order (top-left → bottom-right, row-major, i.e. ascending by `relRow` then `relCol`). Each successfully placed cell pushes a `{ op: 'place', ... }` entry to the undo log, exactly like a normal `place()` call. The paste command is one atomic sim step but multiple undo entries — undoing each individual placement works naturally with repeated `undo` commands.
- **Snapshot addition:** `Snapshot` gains `clipboard: { relCol: number; relRow: number; type: ModuleType; dir: Dir }[]` (empty array when nothing is copied). The main thread uses this to render the ghost preview. The sim never exposes the absolute positions — the main thread computes them from `originCell + relRow * w + relCol`.

- [ ] **3i: Add `copyBlueprint()` method** (after `undo()`):
```ts
  copyBlueprint(cells: number[]): void {
    if (cells.length === 0) { this.clipboard = []; return; }
    // Find bounding box
    const cols = cells.map((c) => c % this.w);
    const rows = cells.map((c) => Math.floor(c / this.w));
    const minCol = Math.min(...cols);
    const minRow = Math.min(...rows);
    this.clipboard = [];
    // Iterate cells sorted row-major (ascending row, then col) for stable order
    const sorted = [...cells].sort((a, b) => {
      const ra = Math.floor(a / this.w), rb = Math.floor(b / this.w);
      if (ra !== rb) return ra - rb;
      return (a % this.w) - (b % this.w);
    });
    for (const c of sorted) {
      const m = this.modules.get(c);
      if (!m) continue; // empty cells in the selection are skipped
      this.clipboard.push({
        relCol: (c % this.w) - minCol,
        relRow: Math.floor(c / this.w) - minRow,
        type: m.type,
        dir: m.dir,
      });
    }
  }
```

    if (this.clipboard.length === 0) return;
    const originCol = originCell % this.w;
    const originRow = Math.floor(originCell / this.w);
    // Entries already sorted row-major from copyBlueprint; process in that order

## Self-review notes (author)

**Scope coverage:**
- Erase refund (100%) — implemented in `remove()` via `ERASE_REFUND` constant (C2). Tunable in one place.
- Undo (32-entry bounded log) — lives entirely in `World`, deterministic (C2). Main thread sends `{ type: 'undo' }` (C3); Ctrl+Z / Cmd+Z shortcut.
- Blueprint copy/paste — `clipboard` on `World` (C2); copy from cells array; paste iterates row-major, skips occupied/locked/unaffordable, deducts per-cell, pushes undo entries. Main thread handles two-corner selection drag (C3); Blueprint tool in hotbar (C4).

**Determinism:** All three features are pure sim commands. The undo log and clipboard are `World` fields reset by `loadDemo`/`loadSave`. No wall-clock in `src/sim/*`. Order-stable cell processing preserved (`sorted` array in `copyBlueprint`, ascending cell index in `paste` clipboard iteration which was sorted at copy time).

**Single source of truth:** `BUILD_COSTS` from `data.ts` is used for refund and paste cost checks — never duplicated.

**Undo symmetry:**
- `place()` logs `{ op: 'place' }`. Undoing a place removes + refunds the full cost.
- `remove()` logs `{ op: 'remove' }`. Undoing a remove restores the module + takes back the refund.
- Both are symmetric with respect to inventory: net effect of place→undo(place) = zero inventory change.
- Net effect of remove→undo(remove) = zero inventory change.
- Paste logs one `{ op: 'place' }` per successfully placed cell; each can be undone individually.

**Known sequencing:**
- C1 leaves the build red (snapshot missing `clipboard`) until C2 Step 3k. Plan to complete C1+C2 before checking the build.
- C3 extends the `tool` variable type; if TS errors on `'blueprint'` being unassignable, C4 Step 1 (extending `Tool`) may need to land first or be done together with C3.
- C3 and C4 can be done in either order or together; they modify disjoint parts of the codebase.

**Deferred / follow-up work:**
- Multi-cell blueprint ghost preview (renderer-side `setBlueprintGhost` API) — deferred to a future renderer task (C3 uses a single-cell indicator at origin for now).
- Blueprint export/import string (copy to clipboard, share between sessions) — ROADMAP Horizon 5; architecture is compatible (clipboard is serializable `{ relCol, relRow, type, dir }[]`).
- Save v3: `undoLog` and `clipboard` are transient (reset on load); they do NOT need to be persisted and should NOT be added to `SaveState`.
- The `placeRawPublic` test-scaffolding method is a minor code smell; it can be removed and tests restructured to use direct field assignment (`world.clipboard = [...]`) for setup that bypasses unlock/cost checks, which is already done in the unaffordability test variant.
